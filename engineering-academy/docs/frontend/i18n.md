# Internationalization — `src/i18n/`

Dhandho ships UI translations for four languages, matched to where its customers actually operate — small manufacturers, distributors, and retailers across Western and Northern India:

| Code | Language | Native label | File | Lines |
|---|---|---|---|---|
| `en` | English | English | `en.json` | 242 |
| `hi` | Hindi | हिन्दी | `hi.json` | 242 |
| `gu` | Gujarati | ગુજરાતી | `gu.json` | 242 |
| `mr` | Marathi | मराठी | `mr.json` | 243 |

There is no `react-i18next`, `react-intl`, `formatjs`, or `lingui` in `package.json`. The entire i18n system is one file, `src/i18n/index.tsx` (under 100 lines), plus four flat JSON dictionaries.

## The data shape: nested JSON, dot-path lookup

```1:19:src/i18n/en.json
{
  "nav": {
    "dashboard": "Dashboard",
    "sales": "Sales Entry",
    "distribution": "Distribution",
    ...
  },
  "common": {
    "save": "Save",
    "cancel": "Cancel",
    ...
  },
  ...
}
```

```43:54:src/i18n/index.tsx
function lookup(dict: Translations, key: string): string {
  const parts = key.split('.');
  let result: unknown = dict;
  for (const part of parts) {
    if (result && typeof result === 'object') {
      result = (result as Record<string, unknown>)[part];
    } else {
      return key;
    }
  }
  return typeof result === 'string' ? result : key;
}
```

Call sites look like `t('nav.sales')` or `t('common.save')` — a dot-delimited path walked one segment at a time through the nested object. If any segment is missing (a key that exists in `en.json` but was never translated in `mr.json`, say), `lookup` returns the **key itself** as the displayed string, not an empty string or a crash. Seeing `"nav.sales"` literally rendered in the UI is an obvious, self-diagnosing bug report — a translator or engineer immediately knows both *that* something is missing and *which* key to add, without needing a separate "missing translation" logging system.

> [!NOTE]
> **Why nested JSON with dot-path lookup instead of a flat key-value map (`"nav.sales": "Sales Entry"`)?** Both are valid; this project chose nested for **file readability during translation work**. A translator (often not an engineer) opening `hi.json` sees a natural grouping — all `nav.*` keys sit together, all `common.*` keys sit together — which is easier to review section-by-section than a flat alphabetically-scattered list of `"nav.sales"`, `"common.save"`, `"nav.dashboard"` keys.

## `TypeScript` as the translation-completeness checker

```1:6:src/i18n/index.tsx
import en from './en.json';

type Translations = typeof en;
type Lang = 'en' | 'hi' | 'gu' | 'mr';
```

`Translations` is inferred directly from the **shape of `en.json`** via `typeof`. This is a small but genuinely clever trick: because `hi.json`/`gu.json`/`mr.json` are typed as `Promise<{ default: Translations }>` (via the dynamic import signature below), TypeScript will flag a compile error if a locale file's JSON structure doesn't match English's shape — e.g., if a key was renamed in `en.json` but the corresponding rename was forgotten in `mr.json`. It does **not** catch a missing individual string value (JSON doesn't carry "this string is empty" as a type error) — only structural mismatches. The runtime `lookup()` fallback above is what covers the remaining gap.

## Lazy-loading: English ships in the bundle, the rest don't

```8:14:src/i18n/index.tsx
/** English ships in the main bundle; other locales load on demand. */
const localeLoaders: Record<Exclude<Lang, 'en'>, () => Promise<{ default: Translations }>> = {
  hi: () => import('./hi.json'),
  gu: () => import('./gu.json'),
  mr: () => import('./mr.json'),
};

const cache: Partial<Record<Lang, Translations>> = { en };
```

```60:83:src/i18n/index.tsx
export function LanguageProvider({ children }: { children: React.ReactNode }) {
  const [lang, setLangState] = useState<Lang>(getStoredLang);
  const [dict, setDict] = useState<Translations>(en);

  useEffect(() => {
    let cancelled = false;
    if (lang === 'en') { setDict(en); return; }
    const cached = cache[lang];
    if (cached) { setDict(cached); return; }
    localeLoaders[lang]()
      .then(mod => { if (!cancelled) { cache[lang] = mod.default; setDict(mod.default); } })
      .catch(() => { if (!cancelled) setDict(en); });
    return () => { cancelled = true; };
  }, [lang]);
  ...
}
```

`import en from './en.json'` at the top of the file is a **static import** — Vite bundles it into whatever chunk `index.tsx` ends up in, so it's always available with zero latency. The other three are **dynamic imports**, each becoming its own small chunk that only downloads the first time a user actually switches to that language. Once fetched, a locale is cached in a module-level object (`cache`) so switching back and forth between languages within a session never re-fetches.

> [!TIP]
> **Why is this the right default for this product?** Every tenant is created with English as the default (`getStoredLang()` falls back to `'en'` if nothing is stored), and language-switching is an explicit user action via Settings (`multiLanguageEnabled` gate — some pricing plans don't include it, see `plans` seed data in `server/pg-db.ts`), not something every user does. Shipping three extra locale bundles nobody-yet-asked-for to every single user would waste bytes against the [256 KB gzip budget](../performance/bundle.md) for the common case, in exchange for saving one small network request for the minority who do switch languages.

## `LanguageProvider` and `useTranslation()` — the Context surface

```31:41:src/i18n/index.tsx
interface LangContextValue { lang: Lang; setLang: (l: Lang) => void; t: (key: string) => string; }
const LangContext = createContext<LangContextValue>({ lang: 'en', setLang: () => {}, t: key => key });
```

```95:97:src/i18n/index.tsx
export function useTranslation() {
  return useContext(LangContext);
}
```

The default context value (`t: key => key`) is a safe fallback for any component that somehow renders outside a `LanguageProvider` — it degrades to "just show the raw key," consistent with `lookup()`'s missing-key behavior, rather than throwing. In practice `LanguageProvider` wraps the whole app early enough (in `main.tsx`, outside `App.tsx` even) that this fallback is mostly theoretical, but it's cheap insurance for a Context that many other Contexts in this codebase don't bother with (compare to [session-state.md](./session-state.md)'s `ToastContext`, which has the same defensive default pattern).

`setLang` persists the choice immediately:

```85:88:src/i18n/index.tsx
const setLang = useCallback((l: Lang) => {
  localStorage.setItem(LANG_KEY, l);
  setLangState(l);
}, []);
```

Note this key (`dhandho_lang`) is **not** tenant-scoped the way `session.ts`'s keys are (see [session-state.md](./session-state.md)) — language preference is treated as a *device* preference, not a *tenant-session* one, so switching companies in another tab doesn't reset your chosen language.

## Where translations are actually used

`t('nav.x')` calls are concentrated in navigation labels (`App.tsx`'s sidebar — see [app-shell.md](./app-shell.md)) and shared UI chrome (buttons: save/cancel/delete, common status words). Feature-specific business copy inside individual views (e.g., form field labels deep inside `AccountsView`'s GSTR reconciliation flow) is **not** uniformly translated — the dictionaries top out at ~240 keys covering navigation and common actions, not a full translation of every string in an 18-module ERP.

> [!CAUTION]
> This is a real, visible trade-off rather than an oversight: **full i18n coverage of every screen would be a large, ongoing translation and maintenance cost** for a small team, for a feature (`multiLanguageEnabled`) that's a nice-to-have on top of a product whose primary audience can generally read English business terminology (GST, IRN, e-Way Bill are English/acronym terms even in Hindi/Gujarati/Marathi business contexts). The pragmatic choice was to translate the parts of the UI a non-English-literate shop floor worker interacts with constantly (navigation, common actions) and leave deeper admin/compliance screens in English.

## Trade-offs

| Choice | Benefit | Cost |
|---|---|---|
| Hand-rolled Context + flat JSON instead of `react-i18next`/`formatjs` | No dependency weight, full control over lazy-loading strategy, trivial to understand end-to-end | No pluralization rules, no ICU message format, no interpolation helpers — any of that must be hand-written if ever needed |
| `typeof en` as the translation type | Free structural-completeness checking at compile time | Doesn't catch empty/wrong translations, only missing/extra keys |
| Partial coverage (nav + common only) | Keeps translation effort proportional to actual daily-use surface | Deeper screens remain English-only even with a non-English locale selected |
| Device-scoped language preference (not tenant-scoped) | A shared device's chosen language persists sensibly across different tenant logins | Two tenants on the same device can't each "remember" their own preferred language |

## Exercise

1. A new key `"nav.payroll"` is added to `en.json` but the corresponding key is forgotten in `mr.json`. Describe exactly what a Marathi-language user sees in the sidebar for that nav item, and what mechanism produces that specific output.
2. Why is `import en from './en.json'` a static import while the other three locales use `() => import('./xx.json')`? What would change, bundle-wise, if all four were static imports?
3. Suggest one lightweight addition (without adopting a full i18n library) that would let this system support basic pluralization (e.g., "1 item" vs. "3 items") if a future feature needed it.

<details>
<summary>Answers</summary>

1. `lookup(dict, 'nav.payroll')` walks `dict.nav.payroll`, finds `undefined` (the key doesn't exist in the Marathi dictionary), and returns the original key string `'nav.payroll'` unchanged — so the sidebar literally displays the text "nav.payroll" instead of a translated label. It's an ugly but self-explanatory failure mode.
2. English is guaranteed to be needed by every single visitor (it's the default and the fallback for failed locale loads), so bundling it statically means zero latency for the common case. If all four were static imports, all four ~240-key JSON files would be included in the initial bundle for every user, even the ones who never switch languages — wasted bytes against the bundle budget for three locales most sessions never use.
3. Add a small helper like `tn(key, count)` that looks up `${key}.one` / `${key}.other` (or a simple `count === 1 ? dict[key].one : dict[key].other` convention) inside the existing `lookup()`-based dictionaries — no new library needed, just a naming convention for keys that need count-based variants, plus one new function alongside `t`.
